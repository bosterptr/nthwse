Methodology for Input Validation Testing in Web Applications.

INTRODUCTION

The article describes the methodology for Input Validation Testing in web applications in the various application fields.

This article will be a general introduction to input validation testing, the next ones will describe specific vulnerabilities, e.g.,. XSS, SQLi.

The advice in this article is based on the following:

  • OWASP Web Security Testing Guide
  • OWASP Application Security Verification Standard
  • NIST recommendations
  • Bug bounty reports
  • Portswigger Academy
  • Own experience.

TOOLING

Many different tools can help you test the Input Validation. The primary tool is the Burp Suite Pro with additional extensions. Additionally, there are plenty of standalone tools & scanners that you can use. I aggregated most of them in my CRIMSON project.

II. ANALYZE THE VALID DATA FLOW

The data flow analysis evaluates how data is transmitted, processed, and stored within the web application. This can be achieved by examining the web application's inputs(sources) and outputs(sinks). Some techniques for analyzing data flow in the web application include:

  • Investigating how the system handles valid data.

Sometimes you cannot see the results of the process engine directly within the web application source code, but there could be potential blind interactions (SSRF, blind SSTI/XSS/SQLi/RCE vulnerability) where you should use the collaborator for testing. For instance, for the below url parameter value:

  • Analyze how data is transmitted between the client and server, including encryption and decryption. Is it possible to sniff or tamper it?
  • Analyze how data is stored, including databases, files, and logs. Determine if unauthorized users can access or modify it.
  • Analyze how the application handles data outputs.
    What is the context of the stored/reflected data?

During the data flow analysis with a valid data, you should also check if there are any sensitive data disclosed. This step will be double-checked later on.

After taking these steps, you should have an overall idea of how the web application works, what are the entry points that you can test and what is the data flow of the input.

III. TEST FOR DIFFERENT INPUT SCHEMAS (MIME type)

The default schema (Content-Type header) for communication with the application triggered during normal usage could be changeable.

The example Content-Type header values:

application/x-www-form-urlencoded
application/json
multipart/form-data
application/xml

It may be that the application usually accepts data in the application/x-www-form-urlencoded format and no vulnerability occurs. However, after changing the content-type to application/json, it will accept data in JSON format and the vulnerability will appear.

IV. DEFINE THE VALID DATA FOR EACH INPUT FIELD (source)

To conduct the Input Validation Testing, you must first define the correct type, format, and boundaries for your payloads for each input field.

  • Data types — identify the correct data type for each input field.

  • Data formats — identify the correct format of data for each input field.

There could be many different formats encapsulated in each other. For instance there could be the Upload functionality, which accepts only the base64 encoded file in one of the JSON parameter value. The example is shown below:

Another thing to consider in the example above is the JSON format. You should escape the double-quote sign(“) in all payloads, to not jump off the input field that is being the object of testing.

  • Data boundaries — identify the correct range for each input field.

Focus not only on the length of the strings, but also the minimum and maximum number that the API accepts. Sometimes there could be some business logic flaws.

After this step, you are ready to prepare payloads for each input field, that can pass through the initial validation and reach the core function. You should always start from this.

V. PREPARE THE PAYLOADS

The payload wordlist should cover at least 5 test cases:

  1. Valid malicious values — payloads designed to test the system's ability to defend against different attack vectors. These inputs conform to the system's constraints (expected type, format, and length). The system should accept and process these payloads correctly (sanitize & parametrize).
  2. Invalid malicious values — payloads designed to test the system's ability to defend against different attack vectors. These inputs do not conform to the system's constraints (expected type, format, and length). These inputs help test the system's ability to handle inconsistent or incorrect data and should be rejected by the system and not processed.
  3. Boundary values — the limits of the expected range of inputs. These inputs help test the system's behaviour when dealing with maximum or minimum values and inputs within or outside the defined range.
  4. Null inputs — null/nil/Noneor no value at all. These inputs help test the system's behaviour when dealing with missing or empty inputs.
  5. Large inputs — values larger than the expected size or length.

Additionally, the payload wordlist should consist of the following:

  • Special characters, such as <,>,’,” to check if they are properly sanitized and encoded.

Fell free to use my single-char wordlist, which covers a lot of edge special characters that could trigger an undesired behaviour in the application.

  • Out-of-band payloads to test for invisible vulnerabilities such as blind SQLi/SSTI/XSS/RFI or SSRF.

The example OOB wordlist from the crimson repository.

  • Serialized payloads — depend on the programming language used to build the application.

To simplify the payload generation process across various applications, you can use the SerializedPayloadGenerator.

  • Multi-Encoding Byte Wordlist Generator:
perl -e 'binmode(STDOUT); print map { chr($_)."\n" } 0..255;' > one_byte_fuzz.bin
perl -e 'binmode(STDOUT, ":utf8"); print map { chr($_)."\n" } 128..255;' >> one_byte_fuzz.bin
perl -e 'binmode(STDOUT, ":encoding(UTF-16LE)"); print map { pack("v*", $_)."\n" } 0..255;' >> one_byte_fuzz.bin
perl -e 'binmode(STDOUT, ":encoding(UTF-16BE)"); print map { pack("n*", $_)."\n" } 0..255;' >> one_byte_fuzz.bin
perl -e 'binmode(STDOUT, ":encoding(ISO-8859-1)"); print chr for 0..255;' >> one_byte_fuzz.bin
perl -e 'binmode(STDOUT, ":encoding(UTF-32LE)"); print map { pack("V*", $_)."\n" } 0..255;' >> one_byte_fuzz.bin
perl -e 'binmode(STDOUT, ":encoding(UTF-32BE)"); print map { pack("N*", $_)."\n" } 0..255;' >> one_byte_fuzz.bin

This Perl script generates a comprehensive wordlist that includes bytes in various Unicode encodings. It uses different Perl functions to create byte representations for values ranging from 0x00 to 0xFF and adds them to a file named one_byte_fuzz.bin. This resource is valuable for testing applications and systems that handle (or not :D) multiple-byte encodings.

After creating the payloads, it is finally time for a testing.

VI. ANALYSE THE PAYLOAD FLOW (Input Validation Testing)

Check how the application handles the payloads.

This step is similar to the II. ANALYSE THE VALID DATA FLOW, but instead of checking only the flow of the data, examine if any payload triggered vulnerability in any sink.

Make sure you checked all of the data sinks, there could be a second order vulnerabilities if data is stored somewhere in the application.

  • Improper Input Validation — pay special attention to encoding & sanitization & parametrization issues (CWE-20).
  • Error messages — verify that the application displays meaningful error messages for invalid inputs, does not reveal sensitive information, and that the user cannot modify the error page's content (CWE-79).

For instance http://afine.com/login?error=CUSTOM_ERROR_TEXT
