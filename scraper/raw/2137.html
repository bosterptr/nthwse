Application Security Testing for XML eXternal Entity injections.

INTRODUCTION

The article describes how to test the application to find XXE injection vulnerabilities. The advice in this article is based on the following:

  • OWASP Web Security Testing Guide
  • OWASP Application Security Verification Standard
  • Bug bounty reports
  • Own experience.

GUIDELINES

In the below guidelines, I assume that you identified the application entry points described in the AppSec Tales XI | Input Validation:

XXE injections can be found in any part of an HTTP body. However, they often occur in POST request with Content-Type: application/xml .

I. XML IDENTIFICATION — CONTENT & SCHEMA

Begin with a basic XML request to see how the application processes it.

  • Use a basic XML format without external entities and content type specifications to ensure correct handling by the application.
  • Test for XML schema to identify additional input points if the proxy request uses a different schema.
  • According to RFC 7303, it is recommended to use application/xml.
  • However, in some scenarios, text/xml or omitting the Content-Type header could work. Below is a small wordlist for testing this:

application/xml

application/xhtml+xml
application/rss+xml
application/atom+xml
application/svg+xml
application/mathml+xml
application/opensearchdescription+xml
application/vnd.google-earth.kml+xml
application/vnd.oasis.opendocument.text
application/vnd.oasis.opendocument.spreadsheet
application/vnd.oasis.opendocument.presentation
application/vnd.openxmlformats-officedocument.wordprocessingml.document
application/vnd.openxmlformats-officedocument.spreadsheetml.sheet
application/vnd.openxmlformats-officedocument.presentationml.presentation

Once you determine where the application accepts XML input and identify a proper request handled by the application, go to the next point.

II. ERROR-BASED XML IDENTIFICATION

Intentionally trigger an error and then analyze the response body.

  • Check how the target handles the special characters as described in:

III. DoS XXE DETECTION — BILLION LAUGHTS ATTACK

Detect XXE by using Billion Laugh Attack.

  • In the below payload, the &a10; is the final entity, which resolves to an overlong dosdosdosdosdos… string that contains a total of 10¹⁰ “dos”.
<!DOCTYPE data [
<!ENTITY a0 "dos" >
<!ENTITY a1 "&a0;&a0;&a0;&a0;&a0;&a0;&a0;&a0;&a0;&a0;">
<!ENTITY a2 "&a1;&a1;&a1;&a1;&a1;&a1;&a1;&a1;&a1;&a1;">
<!ENTITY a3 "&a2;&a2;&a2;&a2;&a2;&a2;&a2;&a2;&a2;&a2;">
<!ENTITY a4 "&a3;&a3;&a3;&a3;&a3;&a3;&a3;&a3;&a3;&a3;">
<!ENTITY a5 "&a4;&a4;&a4;&a4;&a4;&a4;&a4;&a4;&a4;&a4;">
<!ENTITY a6 "&a5;&a5;&a5;&a5;&a5;&a5;&a5;&a5;&a5;&a5;">
<!ENTITY a7 "&a6;&a6;&a6;&a6;&a6;&a6;&a6;&a6;&a6;&a6;">
<!ENTITY a8 "&a7;&a7;&a7;&a7;&a7;&a7;&a7;&a7;&a7;&a7;">

<data>&a10;</data>
  • If triggered, the server would crash or respond after a very long time.

When checking for XXE, I start testing from DoS since it is likely that the vulnerable app is also susceptible to Billion Laughs. However, DoS techniques may not be possible in some situations, especially in production environments.

IV. OAST XXE DETECTION — SSRF

Use the out-of-bounds technique and observe any interactions.

  • Sometimes, the application may not display content directly, and you cannot use the DoS technique.
  • For such scenarios, check if the XXE worked by using SSRF payloads.

  • Additionally, use the payload without the % sign before because parameter entities may be blocked while general entity declarations (without the % sign) may not be.

When testing for XXE, I usually begin with OAST as it helps me uncover any vulnerabilities that might not be immediately apparent but exist on the server backend. Moreover, this approach might lead to errors appearing directly in the response body, which is a double advantage.

V. BASIC ENTITY TEST

Use the simplest payload to check if an entity is evaluated and reflected.

  • Entities are defined inside the DOCTYPE element.
<!DOCTYPE replace [<!ENTITY basicXXE "CrimsonXXE"> ]>
 <creds>
  <username>&basicXXE;</username>
 </creds>

You should see CrimsonXXE in the response.

VI. FILE RETRIEVAL

Use entity to retrieve file content from the filesystem.

  • Check XXE_manual, which contains various payloads for file retrieval:

If file retrieval does not work, check directory listening.

VII. XInclude

Check if the server embeds client data in XML on the backend.

  • Suppose you encounter an application incorporating client-submitted data into an XML document and parsing it server-side.
  • In this case, you can use XInclude:
<foo xmlns:xi="http://www.w3.org/2001/XInclude"><xi:include parse="text" href="file:///etc/hosts"/></foo>

You can practice it in the Burp Suite Academy lab.

IMPACT

Usually, XXE injection leads to information disclosure through a file read and may cause Denial of Service due to a Billion Laughs Attack.

For further exploitation guidance, refer to PayloadsAllTheThings.

FINAL WORDS

Checking off each item on this list will certainly help ensure that no straightforward vulnerabilities are overlooked during testing.
