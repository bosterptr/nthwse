Autoaudit — A Linux Forensics Investigation Tool

Adam Messer

·

Follow

Autoaudit is a Linux forensics investigation tool designed to identify evidence of log tampering and identity attacks in the /var/log/wtmp, /var/log/btmp, and /var/run/utmp logs. It is lightweight, extensible, and intuitive — written in Bash (shell), it can run natively on any Linux system without dependencies, and it can be utilized as either a single-use forensics tool or part of a continuous monitoring solution.

Autoaudit began as an exercise to develop my Bash scripting skills following completion of the CompTIA Linux+ certification. My initial intent was to complete a personal project that would get me some chops writing, troubleshooting, and executing scripts in Bash as well as with general Linux security and administration. You can read about my experience, lessons learned, and the background of Autoaudit in a subsequent article. This article, however, will focus on the utility and purpose of this tool.

The Function of Autoaudit

Autoaudit is designed to rapidly detect log tampering and identity attacks in the wtmp, btmp, and utmp log sources in Linux systems. These logs can be vital in a defender’s hunt to illuminate an adversary who has accessed (or attempted to access) a Linux host. Consequently, manipulating the *tmp logs is a common tactic for someone trying to cover their tracks after an intrusion. For those unaware, here’s a bit of background on each of these logs.

  1. /var/log/wtmp
    - The wtmp file records user login and logout activity. It stores a history of login sessions, including the username, terminal, IP address, and login/logout times. This log is crucial for auditing user activities on a Linux system.
    - Entries are added to the wtmp file when a user logs in or out of the system. It is continuously updated to maintain a record of user sessions.
  2. /var/log/btmp
    - The btmp file is responsible for logging failed login attempts. It keeps track of unsuccessful login efforts, providing information about the source of the attempt, the username, and the timestamp.
    - Entries are added to the btmp file each time there is an unsuccessful login attempt, helping administrators monitor and respond to potential security threats.
  3. /var/run/utmp
    -     The utmp file stores information about the current status of user sessions on the system. It includes details such as the username, terminal, login process ID (PID), and the time a user logged in.
    - The utmp file is continually updated as users log in and out of the system. It reflects the real-time status of active user sessions.

Autoaudit works by analyzing all three of these logs for tampering and then querying the btmp file set for evidence of identity attacks like bruteforce attempts.

Log Tampering Detections

Autoaudit will identify a variety of log tampering indicators. This makes it a powerful tool for rapid analysis of the *tmp logs for forensic investigations. Here are the different tampering indicators it’s designed to detect.

  • Abnormal Record Types — this function detects zeroed out logs by checking the record type (first column) in all input log sources. If an adversary conducted a log tampering attempt that zeroed out all fields of a log, the record type would be set to zero. Additionally, if an adversary attempted to manipulate the record type with any non-standard value, this function would alert.
  • Log Erasure — this function identifies emptied fields in the *tmp logs beyond a typical threshold (a value which can be adjusted by the user). An adversary may use hexedit or utmpdump -r to erase all fields within a log entry but not erase the entry itself. This function will alert when it identifies an anomalous number of empty fields in a log entry.
  • Timestamp Manipulation — this function detects manipulated and discontinuous timestamps in a log. The function compares all timestamps in the log and identifies if any are out of order — indicating potential log manipulation. An adversary may manipulate a logged event but not completely remove it in order to evade detection. This function will alert if it identifies a timestamp that has been manipulated and is now inconsistent with other entry timestamps. Additionally, this function will detect if a timestamp has been zeroed out (reset to epoch time in Linux — 1970–01–01 00:00:00).
  • General File Manipulation — this function detects discrepancies between the most recent entry time and file modification time. If an adversary were to manipulate a *tmp file with utmpdump -r or hexedit, then the file modification time would not match up with the most recent entry in the log. Outside of tampering, the file modification time will always match with the most recent log entry. This function will alert if the two timestamps are not equivalent.

Identity Attack Detections

Autoaudit has two methods for identifying bruteforce login attempts against an endpoint.

  • Bruteforce Detection by User — this function detects bruteforce or login attempts by users that are not registered on the system. It compares all failed logins with users found in /etc/passwd and identifies any attempted logins by users who are not registered on the system. This is useful to rapidly identify if credential stuffing or bruteforce attacks are being attempted on a system and their prevalence.
  • Bruteforce by Time — this function detects bruteforce or login attempts based on time entries. With sensitivity variables that can be defined by an admin (sens_num and sens_time), the function will identify suspicious numbers of failed logins within a certain time period. This is effective for rapidly identifying credential stuffing or bruteforce attacks.

Autoaudit in Action

Here’s how to use Autoaudit yourself, and I’ve included an example output from my test endpoint.

How to Use Autoaudit

Autoaudit is designed to be lightweight and free of dependencies. The only prep needed to use it is to download, add run permissions, set parameters within the script, and then run as root. Upon running, the user will be prompted with the ability to query for Log Tampering (1) or Identity (2). Identity Attacks will always query the file set in the LOG_IDENTITY_FILE variable to detect bruteforce attempts and login attacks. Log Tampering will query whichever logs are defined in the LOG_FILES variable within the first lines of the script. By default, this is /var/run/utmp, /var/log/wtmp, and /var/log/btmp. After making a selection (1 or 2), the script will automatically conduct analysis and then print results to the console. This, of course, can be redirected with a ‘>’ and saved as output for enduring analysis or records.

Autoaudit can be leveraged as part of a continuous monitoring solution by running it as a Cron job — an example below:

30 * * * * echo "1" | /home/kali/Desktop/autoaudit.sh > /home/kali/Desktop/test_output.txt 2>&1

A Test Case

A set of canary logs are easy to develop yourself, and I’ve included a copy of them in the tool’s Github page. If you’re looking to do the former (and maybe learn something along the way), follow the below instructions and see Autoaudit in action.

1. Conduct ‘good’ and ‘bad’ login attempts.

This will generate data in the wtmp and btmp log sets. You can do this a multitude of ways — the easiest being su, SSH, telnet, or GUI logins. Generate both successful and failed logins with both real and fake usernames. Here’s an example of what that could look like:

su root
#input the incorrect password
su root
#input the correct password
ssh kali@127.0.0.1
ssh test@127.0.0.1
ssh hacker@127.0.0.1
ssh hacker@127.0.0.1
ssh hacker@127.0.0.1
ssh hacker@127.0.0.1
ssh root@127.0.0.1

For a touch of complexity, if you have SSH enabled, you can use the test_generator_autoaudit.sh script in the tool’s Github repository to generate a large amount of data for the btmp login. This script simply iterates over a list of usernames and fails SSH attempts for a user-defined number of times. This can easily and quickly generate hundreds of failed login attempts.

2. Create a set of dummy logs.

Rather than tamper with your system’s real security logs, it’s a good practice to create a dummy set of canary logs. Here’s my method:

sudo cp /var/log/btmp /var/log/btmp2
sudo cp /var/log/wtmp /var/log/wtmp2
sudo cp /var/run/utmp /var/run/utmp2

Now, we’ll only ever tamper with the *tmp2 logs.

3. Tamper with the logs.

Here’s the fun part. There are multiple ways to get after this, but I’ll cover two.

The first is the text editor method. To start, save your current log contents to a text document.

sudo utmpdump /var/log/btmp2 > /home/kali/Desktop/btmp2_text.txt

Open that up with your favorite text editor (i.e. vim!) and make some edits between brackets. Here’s what I recommend for some easy canaries:

  1. Change the record type (the first column) to a ‘0’ or a two digit number.
  2. Change the timestamp in one of the entries to a random, out of order time.
  3. Erase the username, access method, and both IP addresses in a single entry.
  4. Change one of the timestamps to 0 epoch time (1970–01–01T00:00:00,000000+00:00)

From there, you can save your manipulated logs back into /var/log/btmp2 with the following command:

sudo utmpdump -r btmp2_text.txt > /var/log/btmp2

The second manipulation method involves using your preferred hex editing tool to manually change the hex in a*tmp file. Here, I’ll use the hexeditor to manipulate the btmp2 file further:

sudo hexeditor -b /var/log/btmp2

Once open, I’ll change an entry in the document to further tamper with the log.

A third method would be to find a nifty tool on Github that conducts log tampering for you. Many of them utilize the zero-out method or complete line erasure to eliminate evidence of malicious activity — both methods are detectable by Autoaudit.

4. Set the target logs.

Now that the test case is set up, we can configure Autoaudit to point to the *tmp2 file sets. Open up the script in a text editor and change the LOG_FILES variable to the following:

LOG_FILES=(
 /var/run/utmp2
 /var/log/wtmp2
 /var/log/btmp2
)

Then, change the IDENTITY_LOG_FILE variable to the following:

IDENTITY_LOG_FILE=(
 /var/log/btmp2
)

5. Run Autoaudit

Now, run Autoaudit as with root privileges:

sudo ./autoaudit.sh

Try both methods — 1=Log Tampering Attacks and 2=Identity Attacks. Autoaudit will then run a series of utmpdump commands to generate the output necessary for analysis. But Why?

Why is this tool useful when these manipulations are so easy to see and read? Well, forensic investigators and network defenders may not have the luxury of perusing a ten-line log. Often, a single host may contain hundreds or thousands of lines of log entries across dozens of log files. Autoaudit is designed to be quick and extensible so that a user could query any number of log files and log entries — that’s where Autoaudit is powerful and time-saving. If you’d like to see this in action at scale, on the tool’s Github I’ve included a 200+ line canary file that you can use to test Autoaudit without ever having to manipulate your own logs yourself. See if you can spot the manipulations without the tool and then experience the time-saving potential of Autoaudit.
