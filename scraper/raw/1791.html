Docker and Linux Capabilities

Sometimes the application/tool packaged in the Docker image might need to perform a privileged operation in order to function. For example, it might need to insert a kernel module into the kernel of the Docker host.

In such a case, the Docker allows the user to add an additional Linux capability (i.e. SYS_MODULE) to the container. This approach is way better than using an all-powerful “privileged” option.

For uninitialized, Linux Capabilities are used to allow binaries (executed by non-root users) to perform privileged operations without providing them all root permissions. There are currently 40 capabilities supported by the Linux kernel. The list can be found here.

As this topic is out of the scope of this post, we will encourage the reader to check more on the following links:

  • Linux capabilities in practice

Lab Scenario

We have set up the below scenario in our Attack-Defense labs for our students to practice. The screenshots have been taken from our online lab environment.

Lab: Abusing SYS_MODULE Capability

This lab consists of a Docker host with multiple Docker containers running on it. The user or practitioner will get command-line interface (CLI) access to a bash shell inside a running container, through the web browser.

Here, the assumption is that the attacker (user) has got a Remote Code Execution (RCE) on the container by exploiting the application/web services running inside it. And, then he has managed to upgrade it to a fully functional duplex shell session.

Objective: Break out of the container by abusing the SYS_MODULE capability and retrieve the flag kept in the root directory of the Docker host system!

Solution:

Step 1: Check the capabilities provided to the docker container.

Command: capsh — print

The container has SYS_MODULE capability. As a result, the container can insert/remove kernel modules in/from the kernel of the Docker host machine.

Step 2: Find the IP address of the docker container.

Command: ifconfig

The IP address of the docker container is 172.17.0.2 and the host machine mostly creates an interface that acts as a gateway for the Docker network. And, generally, the first IP address of the range is used for that i.e. 172.17.0.1 in this case.

Step 3: Write a program to invoke a reverse shell with the help of usermode Helper API.

Source Code:

Explanation

  • The call_usermodehelper function is used to create user mode processes from kernel space.
  • The call_usermodehelper function takes three parameters: argv, envp and UMH_WAIT_EXEC
  • The arguments to the program are stored in argv.
  • The environment variables are stored in envp.
  • UMH_WAIT_EXEC causes the kernel module to wait until the loader executes the program.

Save the above program as “reverse-shell.c”

Command: cat reverse-shell.c

Step 4: Create a Makefile to compile the kernel module.

Makefile:

Command: cat Makefile

Step 5: Make the kernel module.

Command: make

Step 6: The idea here is to compile this kernel module (reverse-shell.ko) into the kernel of the Docker host machine. On insertion, it will use usermode helper to create a reverse connect process from the userspace of the Docker host machine.

Start a netcat listener on port 4444 to handle the connect back.

Command: nc -vnlp 4444

Step 7: Copy and paste the lab URL in a new browser tab to open another terminal/console/CLI session. This session is also inside the same container. Now, insert the kernel module using insmod.

Command: insmod reverse-shell.ko

A connect back shell session will be available to netcat listening on port 4444 of the container that will provide a bash shell on the Docker host machine. The module will wait in the same state for the bash session to be closed and only then it will exit.

Step 8: After getting the shell session, list the processes running on the Docker host machine.

Command: ps -eaf

Step 9: Search for the flag file and retrieve the flag.

Commands:

find / -name flag 2>/dev/null

cat /root/flag

Flag: d4974ae46bb7e922703e77497a53d091
