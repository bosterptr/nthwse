Application Security Testing for the Server Side Template Injection.

INTRODUCTION

The article describes how to test the application to find Server Side Template Injection vulnerabilities. The advice in this article is based on the following:

  • OWASP Web Security Testing Guide
  • OWASP Application Security Verification Standard
  • Bug bounty reports
  • Own experience.

TOOLING

Tools with basic usage instructions & wordlist used for the SSTI detection.

STANDALONE TOOLS

  • SSTImap — semi-automated SSTI detection & exploitation tool.

In the current version, it is impossible to use RAW requests from a file and specify the injection point using the * as it goes with SQLmap. However, you can inject at the end of the PATH without using any parameters: https://afine.com/path/

# SINGLE URL - GET PARAMTER
python sstimap.py -u https://afine.com/path?param=value
# SINGLE URL - POST PARAMTER VALUE
python sstimap.py -u https://afine.com/ -m POST -d test=inject
# SINGLE URL - POST PARAMETER NAME (inject_here will be targeted)
python sstimap.py -u https://afine.com/ -m POST -d inject_here=inject -M inject_here
# MULTIPLE URLS
for url in $(cat urls.txt)
do
    echo "[*] TESTING - $url" | tee -a SSTImap.txt
    python3 sstimap.py -u $url | tail -n +17 | tee -a SSTImap.txt
done
# CAPTURED REQUEST & HEADER & SELECTED PATH & SECOND ORDER INJECTION 
I hope functionalities will be provided in the future, for now use Burp.
# PATH INJECTION (works only at the end of the path)
python3 sstimap.py -u "http://afine.com/user/
# ALL-AROUND OPTIONS
## RANDOM USER AGENT TO BYPASS WAF
--random-user-agent
## THOROUGH TESTS
--level 5
## PROXY TO BURP
--proxy http://127.0.0.1:8080
## ADDITIONAL HEADERS (Authorization)
-H "Auth_header: secret1" -H "Cookie: auth2=secret2"
## METHOD
-m POST
## INJECTION MARKER
-M inject_here
  • crimson_templator — automated SSTI detection for massive scope.

At the moment, it works only with GET parameters. It is suitable for mass testing to hunt for low-hanging fruits, but I would stick to the manual testing with Burp Suite with lists presented below for smaller scope.

# USAGE
python3 crimson_templator.py -w urls.txt -c "Cookie: auth1=qwe; auth2=asd;" -H "you_header_1=1" -H "Your_header_2=2"

The tool uses two approaches for finding the SSTI — String and Error based:

# 1. CHECK FOR SSTI STRING 7777 IN RESPONSE TEXT
#   a) ${1111*7}
#   b) <%= 1111*7 %>
#   c) {{1111*7}}
#   d) #1111*7
# 2. CHECK FOR INTERNAL ERROR 500 STATUS CODE IN RESPONSE
#   a) ${7/0}
#   b) {{7/0}}
#   c) <%= 7/0 %>
#   d) ``
#   e) ${{<%[%'"}}%\.

As you can deduce from the above, the tool could be better. As I mentioned, it is just a small helper for a massive scope. You should not rely 100% on it.

BURP SUITE PRO EXTENSIONS

At the moment of writing, no publicly available Burp Suite extensions could help find SSTI vulnerabilities, nor can the Burp Suite automatic build-in scanner handle it properly. However, you can leverage the Input returned in the response (reflected) to search for the SSTI in these reflections.

WORDLISTS

  • SSTI — Wordlist I created through many CTFs and during my daily job. Additionally, it includes ssti.fuzz wordlist from Payload All The Things and Burp Suite Intruder SSTI wordlist (200 payloads total).

The SSTI wordlist contains some blind out-of-bound payloads.
If you want to use them, replace placeholders with your VPS IP and a COLLABORATOR DOMAIN. One thing to mention — they work for port 80.

cat SSTI | sed "s/vps_ip/123.123.123.123/" | sed "s/domain_collab/your_colaborator_com/"

If you do not want to use them, just cut them off from the wordlist:

cat SSTI | grep -v "vps_ip\|domain_collab"
  • Payload All The Things — payloads corresponding to the given engine.

Just to remind you SSTI can be found in any part of the HTTP request.

I. ENUMERATE TEMPLATE ENGINE BY DOCS

Refer to the manual to identify the template engine.

  • If the application is open-sourced or you have access to the documentation, there is the first place where you should start.
  • Thanks to the documentation, you can learn what template engine is in use and how to use it; sometimes even, you can find security considerations that can help you during the exploitation process.

Six hours of fuzzing can save you 5 minutes of reading documentation.

II. ENUMERATE TEMPLATE ENGINE BY PROGRAMMING LANGUAGE

Find out what language the application is written in.

  • This can lead to the Template Engine being used in a specific language.
  • You can use for this purpose the Wappalyzer extension:

However, do not take this for granted because the injection can sometimes happen in some third-party systems written in a different language that returns output to the application. In such cases, you’d miss this case by limiting yourself to only one language’s template engines.

III. ENUMERATE TEMPLATE ENGINE BY STACK TRACE

Look for any errors that indicate the existence of the template engine.

  • This is not limited to errors related to the engine itself. This means that some stack traces in the application can reveal whether a template engine exists.

You could try to trigger an error in the application using fuzzing techniques.

IV. DETECT SSTI BY TEMPLATE ENGINE PROCESSOR ERROR

Trigger an error in a template engine processor.

  • Using SSTI polyglots payload:
${{<%[%'"}}%\.
{{7*7}}${7*7}<%= 7*7 %>${{7*7}}#{7*7}%asset_name%${{<%[%'"}}%\
7/0{7/0}{{7/0}}${7/0}{{7/0}}<%= 7/0 %>@(7/0)[=7/0][7/0][[${77/0}]]
  • By triggering a 0 division error in a template engine processor:
7/0
{7/0}
{{7/0}}
${7/0}
{{7/0}}
<%= 7/0 %>
@(7/0)
[=7/0]
[7/0]
[[${77/0}]]

The list of payloads that triggers the 0 division errors is not extensive.
The SSTI wordlist contains these payloads and polyglots.

V. DETECT SSTI BY TEMPLATE ENGINE PROCESSOR OUTPUT

Trigger calculations or functions in the template engine.

  • Use the SSTI wordlist and observe the output.
  1. Shows the GREP in Burp Suite Intruder.
  2. It shows how many 5929 values were found in the HTTP response.
  3. Shows the processed value in the response.
  4. Shows another anomaly using similar injections.

Despite I showed the two standalone tools in the tooling section. The Burp Suite Intruder is the best tool currently for detecting SSTI during the black-box.

VI. SECOND-ORDER SSTI

Repeat the above steps for any input parsed elsewhere.

  • In the AppSec Tales XIII, you can find point IX. SECOND-ORDER SQLI describes the example scenario and shows the possibility of automation of second-order vulnerabilities using a Python middleware server.

VII. DETECT SSTI BY TEMPLATE ENGINE PROCESSOR — BLIND

Trigger functions in the template engine that interacts with DNS|HTTP.

  • Start the HTTP server on port 80 on your VPS machine using python3 http.server module:
python3 -m http.server 80 2>&1 | tee -a http.logs
  • Start tcpdump on VPS to capture ICMP packets:

It is worth changing any to a interface that you are using for capturing the ICMP packets, for example eth0. It will make the log smaller.

sudo tcpdump -i any icmp -ttttt -w - -U | tee icmp.logs | tcpdump -r -
  • Use the below payloads from SSTI wordlist for injection and observe OOB interactions:
cat SSTI | grep "vps_ip\|domain_collab"| sed "s/vps_ip/123.123.123.123/" | sed "s/domain_collab/your_colaborator_com/"

The list could be more extensive. Feel free to contribute to my GitHub.

VIII. BOMB TESTING

If there is no length limit of the input, use all of the payloads at once.

Copy-paste all of your payloads and check the results.

  • If you cannot use the new line character, remove it using VSCode as described in the AppSec Tales XII | XSS (point XII).
  • Additionally, you can add some “A” chars if there is no change in the server response after copy-paste all of the payloads.

This method can trigger another error, which could change the context and be vulnerable to SSTI.

IX. MASS TESTING

When you have a lot of URLs to test, use CLI tools.

  • This method works only for GET parameters.
  • Using SSTImap:
# MULTIPLE URLS
for url in $(cat urls.txt)
do
    echo "[*] TESTING - $url" | tee -a SSTImap.txt
    python3 sstimap.py -u $url | tail -n +17 | tee -a SSTImap.txt
done
  • Using crimson_templator:
python3 crimson_templator.py -w urls.txt

This way, you can test for low-hanging fruits on a scale.

IMPACT OF SSTI

Server-side template injection vulnerabilities can be extremely dangerous, allowing attackers to execute code on the back-end server and gain access to sensitive data and files.

After detecting the vulnerability, the attacker would identify the Template Engine and the exploit-detected vulnerability to gain RCE.

I described one of the example attack scenarios I found during my daily work in the below article. In this case, executing code on the back-end server was impossible. However, it was possible to turn it off because one of the available engine templates was vulnerable to DoS:
