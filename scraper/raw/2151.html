Guidelines for the crimson_exploit module usage.

INTRODUCTION

This article will describe how to use the crimson_exploit most optimally.
The module is used for finding the vulnerabilities in many URLs.

The crimson_exploit should be used after the manual testing to ensure nothing was left out. The module is good at finding vulnerabilities with the GET method.

CRIMSON EXPLOIT GUIDELINES

You need the dirs.txt and params.txt directories to start using the third module. You can see an example content of the files on the screen above.

These two files are created in the exp/ directory after finishing the execution of the second module — crimson_target.

If you got the files in the current directory, you could start the tool by simply issuing the c_3 command. However, it is advised to use additional flags:

The module will stop and remind you to remove URLs you do not want to test.

OAST

  • -d flag stands for out-of-bound testing with a given collaborator domain.
  • -i flag stands for out-of-bound testing with a given VPS IP address.

The flags are optional, but if you want to test the URLs with out-of-bound techniques, then both flags must be used at once to make the tool work properly.

The tool uses payloads that send the ICMP packets or TCP packets on port 80 when the payloads are triggered (if the potential vulnerabilities were found).

If you are using these flags, you need to start two listeners on your VPS:

FUZZING

  • -x flag stands for fuzzing. The empty value (‘’ ) instruct the module to use the default wordlist /root/tools/crimson/words/bug .

You can use your wordlist for fuzzing the URLs. The default one has around 9000 payloads, so as you can guess, it will take a while. For instance, to test 100 URLs, the tool needs to make 900 000 requests.

CONTINUING THE EXECUTION

As you can see below, I deleted some URLs using VIM and issued the same command to continue the module execution.

The tool will now:

  • combine dirs.txt and params.txt to one file all.txt ;

It is because some tools uses only dirs.txt and the other ones params.txt , but there are also tools that uses both of them (all.txt).

  • test for reflected XSS vulnerabilities;
  • test for log4j and other vulnerabilities using OOB techniques;
  • test for CRLF injections;
  • test for SSTI injections;
  • test for HTTP request smuggling;
  • test for Broken Links;
  • test for SQL injections;
  • test for Prototype Pollutions;
  • test for header injections;
  • fuzzing all URLs using the GET method only;
  • combining all XSS findings in one wordlist (vuln_xss) ready to copy-paste to the browser to confirm the vulnerability.

You can also use crimson_opener to automatically open it in browser:

python crimson_opener.py -o vuln_xss

OMITTED FLAGS

  • -n flag stands for testing each URL with nuclei .

It will take a very long time to test each URL with the nuclei. This is useful when you are testing multiple domains. Conversely, if the tests cover a single host, it is not worth your time.

  • -o flag turns on fuzzing on all URLs to find an Open Redirection.

By default -x flag with the default wordlist is using the same payloads.

THE OUTPUT

After the execution, you should check all additional files created by this module in the current directory:

1. bug_params.txt         - FUZZED dirs.txt
2. bug_dirs.txt           - FUZZED params.txt
3. vuln_xss               - XSS POC's (for opening in browser)
4. nuclei.txt             - NUCLEI SCAN OUTPUT
5. smuggler.txt           - HTTP REQUEST SMUGGLING OUTPUT
6. prototype-pollution.txt- PROTOTYPE POLLUTIONS
7. broken_links.txt       - BROKEN LINKS
8. sqli/                  - SQL INJECTIONS
9. oob.txt                - LOG FOR OAST CHECK 
10. CRLF.txt              - CRLF
11. OR.txt                - OPEN REDIRECTIONS
12. dalfox.txt            - REFLECTED/DOM XSS VULNS
13. SSTI.txt              - SSTI VULNS (crimson_templator)
14. SSTImap.txt           - SSTI VULNS (SSTImap)
15. headi.txt             - HEADERS INJECTIONS

If any of the files is missing, it means that either the flag was not used or the output was empty.

FINAL WORDS

I plan to develop this module in the future to cover more bugs and improve the speed of fuzzing and fine-tuning payloads.

Currently, the module is excellent for the OAST testing and for FUZZING the URLs using the GET method. It quickly catches reflected | dom-based XSS, simpler SQL injections, and SSTI vulnerabilities. However, It takes a long time. Testing 100 URLs takes around 10 hours, so choose your URLs wisely.

If you have ideas on how to improve this module, feel free to cooperate on Github. I hope the tool serves you as much as it does for me!
